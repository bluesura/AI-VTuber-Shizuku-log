# YouTube配信ログからコメント用タイムスタンプ生成プロンプト（生ファイル版）v1

## このプロンプトの目的

YouTube からダウンロードした 4 種類の生ファイル（`.info.txt` / `.sbv.txt` / `.live_chat.txt` / `.comments.txt`）を直接読み込み、YouTube コメント欄に貼付可能なタイムスタンプ付きシーン説明を生成する。

単なる事務的な事象の羅列（議事録）ではなく、配信の熱量や文脈（Vibe）が伝わる魅力的な目次を作成する。特に**「要約」ではなく「予告（Teaser）」**として機能させ、視聴者が「何があったのか」気になって思わずタイムスタンプをクリックしたくなるように誘導することを最大の目的とする。

> **注意：** このプロンプトは `.compiled.md`（統合ログ版）を使用する `youtube_timestamp_comment_prompt_v4.md` とは別シリーズです。  
> 統合ログを作成せず、生ファイルを直接入力とする場合はこちらを使用してください。

---

## 入力ファイルと各フォーマット

以下の 4 ファイルをすべて入力として読み込む。

---

### ① `.info.txt`（動画メタ情報）

```
# 動画情報
タイトル  : （動画タイトル）
投稿日    : YYYY-MM-DD
動画ID    : （ID文字列）
URL       : https://www.youtube.com/watch?v=（ID）

────────
【説明】
────────
（概要欄テキスト）
```

**読み取るべき情報：**
- 配信タイトル・投稿日（出力ヘッダーへの参照用）
- 概要欄のコマンド説明（`@` `#` プレフィックスの意味を理解するために必須）
  - `@` なし = 普通のコメント（ランダムに一つ反応）
  - `@` 先頭 = 順番待ち列に入るコメント（必ず読まれる）
  - `#` 先頭 = しずくにスルーされるコメント（感想・リアクション専用）

---

### ② `.sbv.txt`（字幕・発言ログ）

```
[H:MM:SS]
（発言テキスト）

[H:MM:SS]
🎵 [音楽]

[H:MM:SS]
（発言テキスト）
```

**フォーマット仕様：**
- タイムスタンプ形式：`[H:MM:SS]`（例：`[0:04:43]`）
- タイムスタンプ行の**次の行**が発言内容
- `🎵 [音楽]` はBGMのみで発言なし
- エントリ間は空行で区切られる
- これは**配信者（しずく）の発言**のみを記録したもの

**処理上の注意：**
- `.live_chat.txt` のタイムスタンプ形式（`[M:SS]`）と単位が異なる。相互参照する際は秒数に揃えて照合する。
  - 例：SBV の `[0:04:43]` = チャットの `[4:43]`

---

### ③ `.live_chat.txt`（ライブチャットログ）

```
# ライブチャット一覧
# 動画: （タイトル）
# URL : （URL）
# 件数: （件数）

[-M:SS] @ユーザー名: コメント内容   ← 配信開始前（負の時刻）
[M:SS] @ユーザー名: コメント内容    ← 配信中（正の時刻）
```

**フォーマット仕様：**
- タイムスタンプ形式：`[M:SS]`（例：`[4:43]`）。60 分を超える場合は `[H:MM:SS]` になることもある
- 負のタイムスタンプ（例：`[-5:20]`）は配信開始前の待機中コメント
- 各行：`[タイムスタンプ] @ユーザー名: メッセージ`

**コメントの種別判別（`.info.txt` のコマンド説明と照合）：**

| 行の先頭パターン | 意味 | タイムスタンプ活用上の扱い |
|----------------|------|--------------------------|
| `@ユーザー名: （メッセージが普通）` | 通常のビューワーコメント | 視聴者反応として参照する |
| `@ユーザー名: @（本文）` | 順番待ち列コメント（視聴者からの質問や話しかけ） | 配信者が応じた可能性あり。SBV の返答と照合 |
| `@ユーザー名: #（本文）` | 視聴者の感想・リアクション専用（配信者に拾われない） | 反応の補足情報として参照 |
| **運営者の `#` コメント** | 配信設定・ゲームの進行説明・補足など | シーン理解に非常に有用。例「`#1問目`」「`#正解です！`」「`#次へ行きます`」など |

> **運営者 `#` コメントの特別な扱い：** 開発者や運営が `#` 付きでログの節目・補足・誤り訂正などをリアルタイムに書き込んでいる場合がある（例：`@aki5503: #すみません、答え設定間違えましたｗｗ`）。これは配信の内部進行を知るための重要な手がかりであり、積極的に参照すること。ただし、チャットユーザー名は出力には含めない。

---

### ④ `.comments.txt`（投稿後コメント）

```
# コメント一覧
# 動画: （タイトル）
# URL : （URL）
# 件数: （件数）

[YYYY-MM-DD HH:MM UTC] @ユーザー名（👍 N）
コメント内容
```

**フォーマット仕様：**
- 動画投稿後の通常コメント（ライブ配信中ではない）
- タイムスタンプは動画内の時刻ではなく、コメントが投稿された日時
- 視聴後の感想・評価として参照する（シーンの時刻との対応はない）
- 件数が少ない場合や、罵倒・スパム系コメントのみの場合は無視してよい

---

## タイムスタンプ参照の統合手順

以下の順序で 4 ファイルを統合して読み込み、タイムスタンプ候補を抽出する。

```
Step 1. .info.txt を読む
  → コマンド体系（@/#の意味）を把握し、チャット解釈の基準を確立する

Step 2. .sbv.txt を時系列で読む
  → 配信者の発言・感情・話題の流れをつかむ
  → 🎵[音楽] のみの区間はBGMタイム（発言なし）

Step 3. .live_chat.txt と .sbv.txt を時刻で突き合わせる
  → 発言と同時刻～前後30秒のチャット反応を確認する
  → 「草」「すごい」「!?」「泣いた」「エモ」などの集中区間を特定する
  → 運営#コメントで節目（問題番号・正解発表・次の話題への移行）を把握する

Step 4. .comments.txt を読む
  → 投稿後コメントから、特に印象的だったシーンのヒントを得る（件数が少ない場合はスキップ可）

Step 5. タイムスタンプ候補を優先度で絞り込み → シーン説明を作成 → 複数コメントに分割
```

## タイムスタンプ選別基準

| カテゴリ | 具体的なシグナル | 優先度 |
|---------|----------------|--------|
| **ハプニング・事件** | エラー・バグ・予期しない展開（運営 `#` コメントの訂正・謝罪なども含む） | 高 |
| **感動シーン** | チャットに「泣いた」「エモ」「いい話」など感動コメントが集中 | 高 |
| **笑いどころ・ネタ** | チャットに「草」「ｗ」「マジか」など笑いコメントが集中 | 中 |
| **正解・不正解の盛り上がり** | クイズ・ゲーム系配信での正解発表、チャット欄が沸く瞬間 | 中 |
| **話題の転換点** | 運営 `#` コメントによる「次の問題」「エンディング」などの節目 | 低 |
| **視聴者参加の見せ場** | 順番待ち `@` コメントへの返答、視聴者との特定のやりとり | 低 |

## シーン説明の作成ルール

### ルール1：「要約」ではなく「予告（チラ見せ）」にする【最重要】
- オチ、発言の答え、ハプニングの具体的な結末など、**「それを読んで満足してしまう内容」は絶対に書かない。**
- 視聴者の「好奇心の空白（Curiosity Gap）」を刺激し、タイムスタンプをクリックしたくなる寸止め表現にする。
- 良い例：`[28:30] まさかの爆弾発言でチャット凍りつく`
- 悪い例：`[28:30] 〇〇は嫌いだと発言し爆笑`（オチを書き切っているためNG）

### ルール1補足：「謎かけ→即答え」の連続ネタバレを防ぐ【最重要】
- **「謎かけ」タイムスタンプの直後に「答え」タイムスタンプを置かない。**
  - NG例：`[5:50] チャットを惑わす第1問` → `[6:37] 「牛乳を注ぐ女」スタート`
  - OK例：`[5:50] チャットを惑わす第1問` → `[6:37] ようやく合点がいったチャット欄`
- 連想ゲーム・クイズの答え・ハプニングの顛末を列挙する説明はNG。

### ルール2：熱心なリスナー目線のトーン
- 事務的な議事録ではなく、配信を楽しんでいるファンの目線で書く。
- 「草」「エモい」「限界化」「沼る」など配信カルチャーの語彙を自然に使う。

### ルール3：適切なトピックの粒度を保つ
- 「雑談」「ゲーム開始」など中身が想像できない大雑把な見出しは避ける。
- 些細な出来事を 2〜3 分ごとに並べる細かすぎる見出しも避ける。

### ルール4：簡潔さを優先
- 1 つのシーン説明は **1〜2 行**（最大 30 文字程度）に収める。

### ルール5：チャット反応を活用
- その時刻前後のチャットコメントから意味を読み取り、説明に反映させる。
- 例：SBT が「叩いてアピールします」のみでも、チャットが「痛くないですｗ」で沸いていれば → `謎のアピール方法にチャット突っ込みどころ満載`

### ルール6：重要な人物名は記載
- 複数人が登場する配信の場合、誰のアクション・発言か明記する。

## 出力形式

### 複数コメント時のヘッダーと基本形式

YouTube コメント欄は 1 コメント約 4000 文字が上限。「続きを読む」で省略されるため、**折りたたまれない冒頭部分に最も魅力的なハイライトと注意書きを配置する。**

```
【タイムスタンプ集】① オープニング～〇問目
🌟ハイライト：[MM:SS] （このパートで最も気になるシーンの寸止め説明）
※これは自動文字起こしからAIによって生成したタイムスタンプのため、ネタバレや誤り、時間のズレを含む可能性があります。

[MM:SS] シーン説明
[MM:SS] シーン説明
…

---

【タイムスタンプ集】② 〇問目 ～ 〇問目
🌟ハイライト：[MM:SS] （このパートの目玉シーン）

[MM:SS] シーン説明
…
```

- 注意書きは**最初のコメント（①）の冒頭にのみ**記載する。
- タイムスタンプ形式は `.sbv.txt` のものに合わせ、`[MM:SS]` または `[H:MM:SS]` を使用する。

## 配信時間別タイムスタンプ数ガイドライン

| 配信時間 | タイムスタンプ数の目安 |
|---------|----------------------|
| 30 分以下 | 10〜15 個 |
| 30〜60 分 | 20〜30 個 |
| 60 分以上 | 30〜50 個 |

## 禁止事項

- [ ] オチ・結末・発言の核心など「ネタバレ・完全な要約」を記載しない（チラ見せに徹する）
- [ ] 「謎かけ」直後のタイムスタンプで答えを書く「連続ネタバレ」をしない
- [ ] ログに存在しない事象や発言を推測・捏造（ハルシネーション）しない
- [ ] SBV の発言テキストをそのまま丸写しにしない
- [ ] 「雑談」「ゲーム開始」など中身が想像できない大雑把な見出しを作らない
- [ ] 個人特定につながる内容を記載しない
- [ ] チャットのユーザー名を出力に含めない
- [ ] 説明を 3 行以上にしない
- [ ] YouTube のリンクやスパムコンテンツを紛れ込ませない

---

## 出力品質チェックリスト

- [ ] シーン説明が動画のオチを明かしておらず、「好奇心の空白」を刺激する予告になっているか
- [ ] 「謎かけ」直後に答えになるタイムスタンプを置いていないか（隣接タイムスタンプを前後で読み直して確認）
- [ ] 冒頭ハイライト配置など、YouTube の「続きを読む」UI を考慮した構成になっているか
- [ ] 注意書きが「※これは自動文字起こしからAIによって生成した〜時間のズレを含む可能性があります。」になっているか
- [ ] 配信カルチャーに沿ったリスナー目線のトーン（Vibe）が反映されているか
- [ ] タイムスタンプが大雑把すぎず、細かすぎない粒度か
- [ ] ハプニング・感動・笑いどころなど重要ハイライトが含まれているか
- [ ] すべてのタイムスタンプが `[MM:SS]` または `[H:MM:SS]` 形式か
- [ ] `.sbv.txt` と `.live_chat.txt` の時刻をきちんと突き合わせてシーン説明を作成しているか
- [ ] 運営 `#` コメントを節目・補足として正しく活用しているか（ユーザー名は出力に含めていないか）

## 処理フロー（内部参考）

1. `.info.txt` を読み、コマンド体系（@/#の意味）と配信概要を把握する
2. `.sbv.txt` を時系列で読み、配信者の発言・感情・話題の流れを整理する
3. `.live_chat.txt` を読み、SBV の各発言と同時刻前後のチャット反応を照合する  
   - 運営 `#` コメントで節目（問題番号・正解発表・進行ミスなど）を把握する
   - チャット反応が集中している区間をハイライト候補としてマークする
4. `.comments.txt` を読み、投稿後コメントから印象的なシーンのヒントを補足する（件数が少ない場合はスキップ可）
5. タイムスタンプ候補を優先度で絞り込み、配信時間に応じた個数に調整する
6. 各タイムスタンプに「予告（チラ見せ）」のシーン説明を作成する（リスナー目線・Vibe 重視）
7. 配信構成に基づいて複数コメントに分割する（不要な場合は 1 コメント）
8. 各パートの「ハイライト」を選定し、ヘッダーに配置する
9. 注意書きを付与して出力する
10. チェックリストで品質確認（特に「謎かけ→即答え」の連続ネタバレがないか、隣接タイムスタンプを前後で読み直して確認する）
