# YouTube配信ログからコメント用タイムスタンプ生成プロンプト v2

## このプロンプトの目的

YouTube Log Prompt で生成された統合ログファイル（`.compiled.md`）から、YouTube コメント欄に貼付可能なタイムスタンプ付きシーン説明を抽出し、複数のコメントに分割して出力する。

視聴者がコメント欄のタイムスタンプをクリックすると動画のそのシーンにジャンプできるようにする。単なる事象の羅列ではなく、配信の熱量や文脈（Vibe）が伝わる魅力的な目次を作成する。

---

## 入力と出力

**入力**：
- `.compiled.md`：統合ログファイル（SRT発言 + ライブチャット統合版）

**出力**：
- YouTube コメント欄用の複数コメント（各4000文字程度以下に分割）
- 形式：`[MM:SS] または [HH:MM:SS]` 形式のタイムスタンプ + シーン説明

---

## タイムスタンプ選別基準

統合ログに含まれるすべてのタイムスタンプから、以下のいずれかに該当するものを抽出する。

| カテゴリ | 具体的なシグナル | 優先度 |
|---------|----------------|--------|
| **ハプニング・事件** | 予期しない音声・エラー・バグ等（ポン酢音など） | 高 |
| **感動シーン** | チャットに「泣いた」「エモ」「いい話」など感動コメントが集中 | 高 |
| **笑いどころ・ネタ** | チャットに「草」「ｗ」「面白い」など笑いコメントが集中 | 中 |
| **マシュマロ読み** | マシュマロの開始地点＆内容（特に感動系・珍しい内容） | 中 |
| **キャラネタ・定番表現** | しずくの「勝ちます」「友達総会」など定番キャラネタ | 中 |
| **話題の転換点** | 配信の大きな構成転換（オープニング→マロ読み→エンディング等） | 低 |

---

## シーン説明の作成ルール

### ルール1：文脈（Vibe）を反映する（クリックしたくなる表現）
- 単なる事象の無味乾燥な要約（例：「ゲーム開始」「雑談」「〇〇の話」）は避ける。
- 配信の空気感、熱量、感情の起伏（Vibe）が伝わる、思わずクリックしたくなるテキストにする。
- 良い例：`[15:00] 〇〇について熱く語る` / `[28:30] 〇〇のバグに完全に沼る` / `[42:10] まさかの〇〇登場で大爆笑`
- 悪い例：`[15:00] 〇〇の話` / `[28:30] バグ発生` / `[42:10] 〇〇登場`

### ルール2：適切なトピックの粒度を保つ
- 「雑談」などの大雑把すぎる見出しや、2〜3分単位で些細な出来事を並べる細かすぎる見出しはノイズになるため避ける。
- ひとつの話題の明確な区切りや、印象的なハイライトごとに抽出する（話題の大きな区切り ＋ その間の重要なハプニング・笑いどころ）。

### ルール3：簡潔さを優先
- 1つのシーン説明は **1〜2行**（最大30文字程度）に収める。
- YouTubeコメント欄で後追い視聴者がスマートフォン等で見ても見やすい長さを維持する。

### ルール4：チャット反応を活用
- そのシーンへの視聴者反応（チャットコメント）から意味を読み取り、説明に反映させる。
- 例：SRTが「吹いてあげます」のみでも、チャット反応が「ポン酢音実装」なら、説明は `ポン酢音（涙をなめる音）` のようにする。

### ルール5：ネタバレを避ける
- マシュマロの内容を説明する場合、結末やしずくの回答は明かさない。
- 良い例：`[22:50] 父との別れについてのマシュマロ（エモい）`
- 悪い例：`[22:50] 父が亡くなった話 → しずくが後悔について語る`（回答をバラしている）

### ルール6：重要な人物名は記載
- 複数人が登場する配信の場合、誰のアクション・発言か明記する。
- 例：`[32:18] あき先生の突然のマイクテスト`

## 出力形式

### 基本形式

出力するコメントの冒頭には、以下の注意書きを必ず記載すること。

```
※これは自動文字起こしからAIによって生成したタイムスタンプのため、ネタバレや誤り、時間のズレを含む可能性があります。

[MM:SS] シーン説明（簡潔な1〜2行）
[MM:SS] シーン説明
[MM:SS] シーン説明
…
```

### 複数コメント分割ルール

YouTube コメント欄は1コメント約4000文字が上限。以下の基準で分割する：

- **1つのコメント = 大体20〜30個のタイムスタンプ＋説明**（文字数で4000文字以下）
- 分割地点は**配信の構成上の区切り目**を優先（オープニング→マロ読み開始など）
- 分割が不可避な場合は空行を入れて視認性を保つ

### 複数コメント時のヘッダー

最初のコメント（①）の冒頭にのみ、AI生成に関する注意書きを含めること。

```
【タイムスタンプ集】① オープニング～マロ読み開始
※これは自動文字起こしからAIによって生成したタイムスタンプのため、誤りを含む可能性があります。

[MM:SS] シーン説明
…

---

【タイムスタンプ集】② マシュマロ① ～ マシュマロ③

[MM:SS] シーン説明
…

---

【タイムスタンプ集】③ マシュマロ④ ～ エンディング

[MM:SS] シーン説明
…
```

## 優先度に基づくフィルタリング

### 自動フィルタリング戦略

1. **高優先度**（ハプニング・感動）はすべて採用
2. **中優先度**（笑い・マシュマロ・定番ネタ）は**代表的なもの**を厳選
3. **低優先度**（話題転換）は、全体の文脈をつなぐ上で必要な箇所のみ残し、大雑把なものは省略する

### 配信時間別のガイドライン

| 配信時間 | タイムスタンプ数の目安 |
|---------|----------------------|
| 30分以下 | 10〜15個 |
| 30〜60分 | 20〜30個 |
| 60分以上 | 30〜50個 |

※旧バージョンより目安を減らし、より質の高い（Vibeのある）トピックに絞り込むようにしています。

---

## 禁止事項

- [ ] SRTの発言を丸写しにしない（簡潔で魅力的な説明に変換する）
- [ ] 結末・回答・ネタバレを記載しない
- [ ] 「雑談」「ゲーム開始」など、中身が想像できない大雑把な見出しを作らない
- [ ] 個人特定につながるコメント内容は記載しない（マシュマロ投稿者の個人情報など）
-[ ] チャットのユーザー名を記載しない
- [ ] 説明を3行以上にしない（簡潔さを優先）
- [ ] YouTubeのリンクやスパムプラグを紛れ込ませない

---

## 出力品質のチェックリスト

- [ ] シーン説明が単なる事象の要約ではなく、文脈（Vibe）や感情が伝わるクリックしたくなる表現になっているか
- [ ] 「大雑把すぎる」あるいは「細かすぎる」タイムスタンプになっていないか（適切な粒度が保たれているか）
- [ ] すべてのタイムスタンプが `[MM:SS]` または `[HH:MM:SS]` 形式か
- [ ] 1つのシーン説明が1〜2行（30文字以内）か
- [ ] ハプニング・感動シーンがすべて含まれているか
- [ ] ネタバレ・結末が記載されていないか
- [ ] 複数コメント分割時にヘッダーが付いているか
- [ ] コメント冒頭に「自動文字起こしからAIによって生成した」旨の注意書きがあるか
- [ ] 個人特定情報（ユーザー名・プライベート情報）が含まれていないか

---

## 処理フロー（内部参考）

1. `.compiled.md` を読み込む
2. 全タイムスタンプを抽出 → タイムスタンプごとに発言＋チャット反応を確認
3. トピックの粒度を調整し、細かすぎる事象は大きなトピックに統合する
4. 優先度基準に基づいてフィルタリング
5. 各タイムスタンプに対してシーン説明を作成（文脈/Vibeを反映し、クリックしたくなる表現に）
6. 配信時間に応じたタイムスタンプ数に厳選調整
7. 配信構成に基づいて複数コメント分割（不要な場合は1コメント）
8. ヘッダーおよび注意書き付きで出力
9. チェックリストで品質確認
